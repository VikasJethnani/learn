Pillar 01 — Metrics
Numeric measurements at regular intervals
Metrics are time-series data points: a numeric value, a timestamp, and labels. Think cpu_usage{host="web-01"} 72.4 at 14:03:22. They're cheap to store, fast to query, and ideal for dashboards and alerting. Unlike logs, metrics are pre-aggregated — you lose individual event detail but gain speed and scale.
App Code → SDK → Agent / Collector → Time Series DB → Dashboard
  • App Code — your service instruments key operations using a client library. For example, incrementing http_requests_total on every request, or recording request_duration_seconds as a histogram.
  • SDK / Agent — the Prometheus client library exposes a /metrics HTTP endpoint on your service. Alternatively, an OTel SDK pushes metrics to a collector.
  • Collector — Prometheus scrapes (pulls) the /metrics endpoint every 15s. In push-based setups, an OTel Collector receives and batches the data.
  • Time Series DB — Prometheus stores the data in its built-in TSDB, optimized for append-heavy numeric workloads. Each data point is a (timestamp, value, labels) tuple.
  • Dashboard — Grafana connects to Prometheus as a datasource. You write PromQL queries to build panels, dashboards, and alert rules.
Prometheus exposition format
# HELP http_requests_total Total HTTP requests # TYPE http_requests_total counter http_requests_total{method="GET",status="200"} 1234 http_requests_total{method="POST",status="500"} 12
PromQL query
# Error rate over 5 minutes rate(http_requests_total{status=~"5.."}[5m]) # StatsD format web.request.duration:320|ms|#env:prod,svc:api

Pillar 02 — Logs
Timestamped records of discrete events
Logs are immutable, append-only records of discrete events: errors, warnings, info messages, audit trails. Unlike metrics, logs preserve the full context of each event — the stack trace, the user_id, the request payload. They're the richest data source but also the most expensive to store and query at scale.
stdout / file → Shipper → Buffer (Kafka) → Indexer → Search UI
  • stdout / file — your app writes log lines to stdout (container-friendly) or to log files on disk. Structured JSON is preferred over plain text for machine parsing.
  • Shipper — a lightweight agent like Filebeat, Fluentd, or Vector tails log files or captures stdout, enriches with metadata (hostname, k8s pod), and forwards.
  • Buffer — a message queue like Kafka absorbs spikes and decouples producers from consumers, preventing log loss during traffic surges.
  • Indexer — the backend (Elasticsearch, Coralogix, Splunk) parses, indexes, and stores log events. This is where full-text search and aggregation queries become possible.
  • Search UI — Kibana, Grafana Loki, or the vendor's native UI lets you search, filter, and visualize logs. Typical queries: filter by level=ERROR and service=payment-api in the last hour.

Pillar 03 — Traces
End-to-end request journeys across services
A trace is a tree of spans representing a single request's path through your distributed system. Each span captures a startTime, duration, serviceName, and operationName. The root span starts at entry; child spans represent downstream calls. Traces reveal latency bottlenecks and cross-service failures.
OTel SDK → Context Propagation → Collector → Backend → Trace Viewer
  • OTel SDK — each service integrates the OpenTelemetry SDK. It automatically creates a span for every inbound request and outbound call, capturing timing and metadata.
  • Context Propagation — the SDK injects a traceparent header into every outgoing HTTP/gRPC call. This header carries the trace_id and parent_span_id, stitching spans across service boundaries.
  • Collector — the OTel Collector receives spans from all services, batches them, and exports to one or more backends. It can also sample, filter, and enrich spans.
  • Backend — Jaeger, Grafana Tempo, or a SaaS vendor stores the spans indexed by trace_id. This enables lookups like "show me every span for trace abc123".
  • Trace Viewer — renders the waterfall diagram you see to the right — each bar is a span, nested by parent-child relationship. You can instantly see which service or database call is the bottleneck.

{ "traceId": "abc123def456789", "spanId": "span_001", "parentSpanId": "", "operationName": "POST /api/invoice", "serviceName": "api-gateway", "duration": 342, "tags": { "http.status": 200 } }

Service Discovery: How Prometheus Finds Your Services

Prometheus doesn't wait for services to register themselves. It actively discovers scrape targets by querying infrastructure APIs. This is critical in dynamic environments where pods, containers, and VMs come and go.

kubernetes_sd

Queries the Kubernetes API server for pods, services, and endpoints. Targets are discovered automatically when pods have the annotation prometheus.io/scrape: "true". The most common method in cloud-native deployments.

consul_sd

Watches the Consul service registry. When a service registers or deregisters, Prometheus updates its scrape targets in real time. Common in HashiCorp-based infrastructure.

dns_sd

Performs DNS SRV record lookups to find service instances. Useful when you have a DNS-based service mesh or internal DNS that resolves service names to multiple IPs.

file_sd

Reads a JSON or YAML file listing target addresses. The simplest method — good for static environments or when targets are managed by an external tool that writes to a file.

Example: Kubernetes SD config

# prometheus.yml
scrape_configs:
  - job_name: "kubernetes-pods"
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true

Why it matters: Without service discovery, you'd need to manually list every IP:port in your Prometheus config. In a Kubernetes cluster where pods are ephemeral and IPs change on every restart, manual configuration is impossible. SD solves this automatically.

Breadcrumbs: What Happened Before the Crash

Sentry breadcrumbs are a trail of events that happened before the error occurred. Think of them as a flight recorder for your application.

UI Interactions

Button clicks, navigation, form submissions — captured automatically by the SDK.

HTTP Requests

Every XHR/fetch call your app makes is logged with URL, method, and status code.

Console Logs

Any console.log(), console.warn() calls are captured as breadcrumbs.

Custom Breadcrumbs

You can add your own: Sentry.addBreadcrumb({ message: "User loaded invoice #123" })

Why it matters: Without breadcrumbs, you see the error but not the context. With breadcrumbs, you see the last 100 actions that led to the crash — making debugging dramatically faster.

Fingerprinting & Grouping: How Sentry Deduplicates Errors

When thousands of the same error occur, you don't want thousands of separate alerts. Sentry uses fingerprinting to group identical errors into a single issue.

Default Fingerprinting

Sentry examines the stack trace and groups errors with the same exception type + call path. A NullPointerException at the same line in the same function = one issue.

Custom Fingerprints

You can override grouping: scope.setFingerprint(["payment-timeout"]) forces all payment timeouts into one group regardless of stack trace differences.

Merge & Split

In the Sentry UI, you can manually merge issues that should be together or split issues that were incorrectly grouped.

Why it matters: Good grouping means your issue list reflects real, distinct bugs. Bad grouping means either noise (too many issues) or missed bugs (different errors merged together).

OTel Collector: The Universal Telemetry Agent

The OpenTelemetry Collector is a vendor-agnostic agent that can receive, process, and export telemetry data (logs, metrics, and traces) from any source to any backend.

Why use it instead of Filebeat/Fluentd?

Traditional shippers handle only one signal type (logs). The OTel Collector handles all three — logs, metrics, and traces — through a single agent. This reduces operational overhead from 3 separate agents to 1.

Receivers

The Collector can ingest data from many sources: OTLP (gRPC/HTTP), Prometheus scrape, filelog (tails files like Filebeat), Kafka, syslog, and more. It can collect both application logs (stdout) and infrastructure logs (kubelet, system journals).

Processors

After receiving, data flows through processors: batch (group for efficiency), memory_limiter (prevent OOM), attributes (add/remove fields), filter (drop noisy logs), k8sattributes (enrich with pod name, namespace, node).

Exporters

Processed data is exported to backends: Elasticsearch, Coralogix, Loki, Kafka, or any OTLP-compatible endpoint. You can export the same data to multiple backends simultaneously (fan-out).

Deployment modes

DaemonSet — one per node, collects from all pods on that node (most common for logs). Sidecar — one per pod, closest to the application. Gateway — centralized, receives from other collectors for aggregation.

Infrastructure logs: The OTel Collector's filelog receiver can tail kubelet logs, kube-proxy logs, and container runtime logs from /var/log/pods/ and /var/log/containers/. This covers OOM kills, pod evictions, CrashLoopBackOff events, and other K8s system-level logs that application code doesn't emit.

W3C Trace Context: How Traces Propagate

When Service A calls Service B, a special HTTP header is injected to carry the trace context across the network boundary. This is the W3C Trace Context standard.

traceparent header format

00-{trace_id}-{span_id}-{flags}

Example: 00-abc123def456-span_a_id-01

Version (00)

Always "00" for the current spec version.

Trace ID

A 32-character hex string shared across ALL services in the same request. This is the glue.

Parent Span ID

The span ID of the caller. This creates the parent-child relationship in the trace tree.

Flags (01)

"01" means sampled (record this trace). "00" means not sampled (skip it).

Why it matters: Without this header, each service's spans are isolated. The traceparent header is what stitches all spans from all services into a single trace tree — enabling the waterfall view above.

Escalation Policies: Making Sure Alerts Don't Get Lost

An escalation policy defines what happens when an alert isn't acknowledged. It's a safety net ensuring that critical issues always reach someone.

Level 1 — Slack notification (0-5 min)

The on-call engineer gets a Slack DM and channel notification. Most alerts are acked here.

Level 2 — SMS + push (5-15 min)

If L1 isn't acknowledged, the system escalates to SMS and mobile push notifications.

Level 3 — Phone call (15-30 min)

If still not acked, an automated phone call is placed. Often escalates to a secondary on-call or engineering manager.

Level 4 — Full team page (30+ min)

The entire on-call rotation and leadership are notified. This should rarely happen.

Key concepts: Dedup keys prevent the same alert from triggering multiple incidents. Suppression rules prevent alerts during maintenance windows. On-call schedules rotate responsibility among team members.

CNCF — Cloud Native Computing Foundation

The Cloud Native Computing Foundation is an open-source foundation under the Linux Foundation that hosts and governs cloud-native projects. When a tool is "CNCF-native," it means it's an officially accepted project that follows cloud-native principles and is designed for Kubernetes-centric environments.

Project Maturity Levels

OTel SDK: How Instrumentation Works Inside Your Service

The OpenTelemetry SDK lives inside your service process. It intercepts calls to frameworks, libraries, and I/O operations to automatically generate telemetry data — without you writing instrumentation code.

Auto vs Manual Instrumentation

Auto-instrumentation

Attach a Java agent JAR and it instruments everything automatically — HTTP handlers, DB queries, gRPC, Redis, Kafka, and more. Zero code changes.

java -javaagent:opentelemetry-javaagent.jar \ -jar invoice-service.jar
Manual instrumentation

Use the OTel API to create custom spans, add attributes, record metrics, and emit logs for business-specific operations.

Span span = tracer .spanBuilder("processInvoice") .startSpan(); span.setAttribute("invoice.id", id); span.end();

What Gets Captured Automatically

  • HTTP requests — method, URL, status, duration
  • Database queries — SQL statement, DB type, connection info
  • gRPC calls — service, method, status
  • Kafka — topic, partition, consumer group
  • Redis — command, key patterns
  • JVM metrics — heap, GC, threads, class loading
SDK vs Agent: The "SDK" is the library code. The "Agent" (Java agent JAR) is a packaging mechanism that loads the SDK + auto-instrumentation into your JVM without modifying your code. For Node.js/Python, you use --require or environment variables instead.

OTel Collector: Deployment Modes

The OTel Collector is a vendor-agnostic proxy that receives, processes, and exports telemetry data. How you deploy it depends on your infrastructure.

DaemonSet (recommended for K8s)

One collector per node. Services send OTLP to localhost:4317. Low latency, collects node-level data (kubelet metrics, container logs via filelog receiver).

Sidecar

One collector per pod. Maximum isolation but higher resource usage. Useful for multi-tenant clusters where each team controls their pipeline config.

Gateway

Centralized deployment (Deployment with HPA). All services send telemetry to a single collector cluster. Simplifies config management but adds a network hop.

Core vs Contrib

Core — Minimal set of receivers/processors/exporters maintained by the OTel project. Stable, small binary.

Contrib — Community-contributed components (200+ integrations). Includes kafkareceiver, lokiexporter, k8sattributesprocessor, etc. Most production deployments use contrib.

Multi-Pipeline Architecture

Each signal (metrics, logs, traces) gets its own pipeline with independent receivers, processors, and exporters. Pipelines are isolated — a failure in the logs pipeline doesn't affect metrics or traces.

Loki: Log Storage That Thinks Like Prometheus

Loki is built on a key insight: don't index log content, only index labels. This makes it dramatically cheaper to run than Elasticsearch.

How It's Different from Elasticsearch

Loki
  • Index labels only (namespace, app, pod)
  • Log lines stored as compressed chunks
  • Grep-like filtering at query time
  • ~10x cheaper storage vs ES
Elasticsearch
  • Full-text inverted index on all fields
  • Blazing fast full-text search
  • High storage + memory cost
  • Better for large-scale text search

Storage Architecture

  • Chunks — compressed log data grouped by label set + time window (~1.5MB each)
  • Index — maps label set → chunk reference (stored in DynamoDB, Cassandra, or BoltDB)
  • Object store — chunks stored in S3/GCS/MinIO (cheap, durable, infinite scale)
When to choose Loki over ES: When you already use Prometheus/Grafana and want logs in the same UI. When cost matters more than full-text search speed. When your primary log access pattern is "show me logs for this service in this time range" rather than "search across all logs for this keyword."

TraceQL Search

For searching without a trace ID, Tempo uses TraceQL — a query language that scans Parquet columnar data with bloom filters to narrow down blocks.

// Find slow POST requests with errors { span.http.method = "POST" && duration > 2s && status = error }

Why Parquet?

  • Columnar format — only read columns needed for the query (span name, duration, status)
  • Excellent compression — 5-10x better than JSON/protobuf for trace data
  • Object store native — write once to S3/GCS, never move data
Cost advantage: No index = no index nodes = ~10x cheaper than Jaeger/Elasticsearch for trace storage. The tradeoff is that search (without trace ID) is slower than indexed backends.

Model Context Protocol (MCP)

An open standard for AI-to-tool communication. Think of it as USB-C for AI — one protocol, any tool.

What is MCP?

MCP (Model Context Protocol) is a standard created by Anthropic that lets AI models interact with external tools and data sources through a uniform interface. Instead of hardcoding API integrations, an AI agent discovers available tools at runtime.

Why it matters

Without MCP, every tool needs a custom adapter. With MCP, the AI agent asks "what can you do?" and gets a structured answer. This is powerful for incident response — the system can adapt to whatever tools a customer uses without code changes.

Where Everything Lives

MCP Lifecycle — From Initialization to Tool Call

JSON-RPC 2.0 — The Wire Protocol

MCP uses JSON-RPC 2.0 as its message format. Every message between client and server is a JSON object following this structure.

Request (client → server)

// Every request has an id — the server responds with the same id { "jsonrpc": "2.0", "id": 1, "method": "tools/call", "params": { "name": "search_issues", "arguments": { ... } } }

Response (server → client)

{ "jsonrpc": "2.0", "id": 1, // matches the request id "result": { "content": [{ "type": "text", "text": "..." }] } }

Notification (no id — no response expected)

// notifications/initialized has no "id" field { "jsonrpc": "2.0", "method": "notifications/initialized" }

Error Response

{ "jsonrpc": "2.0", "id": 1, "error": { "code": -32601, "message": "Method not found" } }

Transport: stdio vs SSE

MCP supports two transport mechanisms. The choice affects where the MCP server runs and how it connects.

stdio (Standard I/O)

  • • MCP server runs as a child process
  • • Communication over stdin/stdout pipes
  • • Fast — no network overhead
  • • Local only — server must be on same machine
  • • Used by: Claude Desktop, local dev tools

SSE (Server-Sent Events)

  • • MCP server runs as an HTTP service
  • • SSE for server→client, POST for client→server
  • • Slight network overhead, but remote-capable
  • • Can run anywhere — cloud, customer's infra
  • • Used by: hosted servers, production deployments

For our system: We'll likely use SSE in production so MCP servers can run alongside the observability tools they wrap (in the customer's infrastructure), while our MCP client connects remotely.

Tool Schema — How the AI Knows What to Pass

Every MCP tool declares an inputSchema — a JSON Schema that tells the AI exactly what arguments the tool expects. The AI reads this at runtime and constructs the correct call without any documentation.

Example: search_issues tool

{ "name": "search_issues", "description": "Search Sentry issues by query string", "inputSchema": { "type": "object", "properties": { "query": { "type": "string", "description": "Sentry search query (e.g. is:unresolved)" }, "project": { "type": "string", "description": "Project slug to filter by" }, "sort": { "type": "string", "enum": ["date", "freq", "priority"], "description": "Sort order for results" } }, "required": ["query"] } }

Why this matters: The AI doesn't need Sentry documentation. It reads this schema, understands that query is required and sort accepts "date", "freq", or "priority", and constructs the correct call. If Sentry adds a new parameter tomorrow, the schema updates automatically — no code change needed on our side.

Key Insights

  • • MCP Client lives in OUR system — we control it. MCP Servers live with the TOOLS — vendors or customers control them.
  • • Connection is persistent — initialize once, call tools many times. No per-call auth negotiation.
  • • Tool discovery is runtime, not build-time. Agent adapts to whatever tools are available for each customer.
  • • inputSchema replaces API documentation — the AI reads the schema and knows how to call the tool.

Impact on What We're Building

  • • Our agent code is tool-agnostic — same code works whether customer uses Sentry, Datadog, or both.
  • • Customer onboarding = connecting MCP servers, not writing adapters. Days → minutes.
  • • Direct API calls (Tier 2) are the fallback for tools without MCP servers — not the default architecture.
  • • We need to build/maintain MCP Client runtime and handle transport negotiation (stdio for local, SSE for remote).

The Canonical Data Model

One schema that every adapter outputs — regardless of which tool it wraps.

interface CanonicalEvent { // Identity event_id: string; // UUID, generated by adapter source_tool: string; // "sentry" | "prometheus" | "pagerduty" | ... source_event_id: string; // Original ID from the tool // Classification event_type: "error" | "metric_anomaly" | "incident" | "log_pattern" | "deployment"; severity: "critical" | "high" | "medium" | "low" | "info"; // What happened title: string; // Human-readable summary description: string; // Detailed context error_type?: string; // e.g., "ConnectionTimeoutError" source_location?: string; // e.g., "db.ConnectionPool.acquire" // Where it happened service_name: string; // Normalized service identifier environment: string; // "production" | "staging" | "development" region?: string; // "us-east-1", "ap-south-1" // When it happened first_occurrence: string; // ISO 8601 last_occurrence: string; // ISO 8601 // Impact occurrence_count: number; affected_users?: number; frequency_per_hour?: number; // Status status: "active" | "acknowledged" | "resolved" | "muted"; assigned_to?: string; // Links back to source source_url: string; // Deep link to the tool's UI // Extensible metadata labels: Record<string, string>; // Tool-specific tags preserved raw_payload?: object; // Original response (for debugging) }
Prometheus → CanonicalEvent
event_type: "metric_anomaly"
severity: derived from alert rule severity label
occurrence_count: from metric value
service_name: from metric.service label
error_type: null (metrics don't have errors)
labels: all metric labels preserved
Sentry → CanonicalEvent
event_type: "error"
severity: from level (error → high, warning → medium)
occurrence_count: from count
error_type: from metadata.type
source_location: from culprit
service_name: from project.slug
PagerDuty → CanonicalEvent
event_type: "incident"
severity: from urgency (high → critical, low → medium)
assigned_to: from assignments[0].assignee.summary
status: triggered → "active", acknowledged → "acknowledged"
service_name: from service.summary
error_type: null (incidents don't have stack traces)
Data Model Alternatives & Trade-offs

Why we chose a flat canonical model — and what we gave up.

Our Choice
Flat Canonical Model

One CanonicalEvent interface for all signal types.

  • Pro: LLM parses one consistent schema
  • Pro: Cross-tool correlation is trivial
  • Pro: Fast serialization, simple queries
  • Con: Some tool-specific nuance goes into labels
  • Con: Optional fields can be null for many tools
Alternative
Domain-Specific Models

Separate MetricEvent, ErrorEvent, IncidentEvent.

  • Pro: Rich type-specific data, type-safe
  • Pro: No nullable fields — all fields are meaningful
  • Con: Agent needs multiple parsers per type
  • Con: Cross-type correlation requires union types
  • Con: LLM prompt must describe N schemas
Alternative
Graph / Triple Model

Everything as (entity, relation, entity) triples.

  • Pro: Maximum flexibility and extensibility
  • Pro: Natural fit for knowledge graphs
  • Con: High query complexity (SPARQL-like)
  • Con: Poor LLM prompt efficiency
  • Con: Harder to reason about at a glance
Why flat canonical wins for AI agents: LLMs work best with consistent, predictable schemas. When an agent sees 50 events, it can parse them all the same way — no conditional logic, no schema switching. The raw_payload field is the escape hatch: if an agent needs tool-specific detail (like a Sentry stack trace), it can always access the original response without the adapter losing any data.

Key Insights

  • Adapters are the only place tool-specific knowledge lives — agents never see raw API formats
  • The canonical model must be LLM-friendly: flat structure, consistent field names, human-readable values
  • raw_payload preserves full tool response fidelity — the adapter is lossless
  • Adding a new tool = writing one adapter file, zero changes to agents or the canonical model
  • Optional fields are null (not omitted) — agents always know the shape

Impact on What We're Building

  • Every new tool integration is bounded work: one adapter file, one field mapping
  • AI agents can correlate across tools because all events share the same shape
  • The labels field preserves tool-specific metadata without schema changes
  • source_url enables agents to link back to the original tool for human escalation
  • Swapping tools (e.g., Prometheus → Datadog) only changes the adapter — nothing else in the stack
