Rosetta Stone: GCP/Azure to AWS

Everything you know still applies - just with different names and a few critical mental model shifts

Pillar 1

Accounts ARE Projects

In GCP you have projects as isolation boundaries inside an org. In AWS, each Account IS the isolation boundary. AWS accounts are the equivalent of GCP projects, but they each have their own IAM, billing, and resource namespace.

In practice: You will have dozens - sometimes hundreds - of AWS accounts. This is normal and expected. One account per workload or team is the standard pattern.
GCP: gcloud projects list
AWS: aws organizations list-accounts
Pillar 2

VPCs Are Regional, Not Global

GCP's global VPC is unique in the cloud world. In AWS (like Azure VNets), VPCs are regional. Need cross-region connectivity? You need VPC peering or Transit Gateway.

The #1 networking surprise for GCP folks: You cannot create a subnet in us-west-2 inside a VPC that lives in us-east-1. Each region needs its own VPC.
GCP: gcloud compute networks create --subnet-mode=auto (global)
AWS: aws ec2 create-vpc --region us-east-1 (regional)
Pillar 3

IAM Has No Resource Hierarchy

GCP's IAM inherits down the org/folder/project tree. AWS IAM is account-scoped. Cross-account access uses AssumeRole, not org-level bindings.

Critical distinction: SCPs (Service Control Policies) provide guardrails at the org level, but they only deny, never grant. A permissive SCP does not give anyone access - it just does not block it.
GCP: gcloud projects add-iam-policy-binding
AWS: aws iam attach-role-policy --role-name X --policy-arn Y

Top 5 Gotchas for GCP Veterans

  1. 1 There is no global VPC. Every region needs its own VPC + Transit Gateway attachment. If you are used to GCP's single global VPC with subnets in every region, you need to rethink your network topology. You will have a VPC per region per account (or per workload).
    # Each region, each account - its own VPC
    aws ec2 create-vpc --cidr-block 10.1.0.0/16 --region us-east-1
    aws ec2 create-vpc --cidr-block 10.2.0.0/16 --region us-west-2
  2. 2 IAM policies are JSON documents attached to users/roles/resources - not bindings on a hierarchy. There is no concept of gcloud projects add-iam-policy-binding. Instead, you write an IAM policy document and attach it to a role or user. Resource-based policies (like S3 bucket policies) are separate JSON documents on the resource itself.
  3. 3 S3 bucket names are globally unique (like GCS), but S3 is regional (unlike GCS multi-region by default). When you create a bucket, it lives in one region. For cross-region redundancy, you need to explicitly configure S3 Cross-Region Replication (CRR).
    aws s3api create-bucket --bucket my-bucket --region us-east-1
    # No --location=us or --location=eu like GCS multi-region
  4. 4 AWS has no equivalent of gcloud config set project. You switch accounts via AssumeRole or SSO profiles. Your ~/.aws/config file will have dozens of [profile xxx] entries, each pointing to a different account/role combo. You switch with --profile or AWS_PROFILE=xxx.
    # ~/.aws/config
    [profile prod-admin]
    sso_start_url = https://my-company.awsapps.com/start
    sso_account_id = 123456789012
    sso_role_name = AdministratorAccess
    region = us-east-1

    # Then: aws s3 ls --profile prod-admin
  5. 5 Default VPCs exist in every region. Delete them immediately in enterprise accounts. They have overly permissive networking - public subnets with auto-assign public IP enabled, a wide-open default security group, and an internet gateway already attached. In a governed landing zone, these are a liability.
    # List default VPCs across all regions
    for region in $(aws ec2 describe-regions --query 'Regions[].RegionName' --output text); do
      echo "$region: $(aws ec2 describe-vpcs --filters Name=isDefault,Values=true --region $region --query 'Vpcs[].VpcId' --output text)"
    done

Key Insights

  • ◆ Multi-account strategy IS the isolation model. Think "one GCP project = one AWS account." This is not optional - it is how AWS security and billing boundaries work.
  • ◆ SCPs are your org-level guardrails. They replace GCP Organization Policies and Azure Policy at the top level. Remember: they only deny, never grant. Start with a deny-list approach for prohibited actions (e.g., prevent disabling CloudTrail, prevent leaving the org).
  • ◆ Identity Center is your single pane for SSO. It replaces the per-account IAM user model. Federate Entra ID (or any SAML/OIDC IdP) into Identity Center, then assign permission sets to accounts. This is the only way to do human access at scale.

Impact on Your Migration

  • Your existing AD/Entra investment carries over directly. Entra ID federates into AWS IAM Identity Center via SAML 2.0 or SCIM. Users, groups, and MFA all come along.
  • Network architecture will feel familiar from Azure (regional VPCs = VNets). Subnets are zonal, route tables are explicit, and security groups are stateful - just like NSGs. You will be productive here quickly.
  • GCP's global model will not apply here. No global VPCs, no global load balancers, no global Cloud Run. Plan for regional deployments with explicit cross-region strategies (Route 53 failover, S3 CRR, multi-region EKS).

The foundation everything else builds on - get this right and the rest falls into place

Core Service

Like GCP's org node - it's for governance only. Every other account rolls up under it. Think of it as the single pane of glass for policy, billing, and account provisioning.

GCP: Folders Azure: Management Groups

Nesting is supported but keep it shallow - two or three levels max. Deep nesting makes SCP inheritance hard to reason about.

5 Apply Service Control Policies (SCPs) +

SCPs are guardrails that deny actions across accounts. They never grant permissions - they only restrict what's possible, even for account root users.

Here's a real SCP that restricts activity to approved regions only:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyNonApprovedRegions",
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "eu-west-1",
            "eu-central-1",
            "us-east-1"
          ]
        },
        "ArnNotLike": {
          "aws:PrincipalARN": "arn:aws:iam::*:role/AWSControlTowerExecution"
        }
      }
    }
  ]
}

Gotcha Notice the ArnNotLike condition - this exempts the Control Tower execution role. Without it, Control Tower itself can't manage accounts in other regions. Always exempt automation roles from region-deny SCPs.

GCP parallel This is like Organization Policies with constraints, but JSON-based and attached per OU. Azure parallel Similar to Azure Policy deny effects at the management group level.

Coming from GCP Think of OUs as Folders and Accounts as Projects. The management account is your Org node - keep it clean. SCPs are like Organization Policies but they're JSON-based deny statements. Where GCP has a project-level resource boundary, AWS has an account-level boundary. The isolation is stronger - each account has its own IAM namespace.

Coming from Azure OUs map to Management Groups, Accounts map to Subscriptions. Control Tower is like Azure Landing Zones. SCPs are like Azure Policy deny effects at the management group level. If you've used Azure Blueprints for standardized subscription setup, Account Factory for Terraform is the AWS equivalent.

Common Mistakes to Avoid

  • Don't put workloads in the management account. Ever. It has elevated privileges across the entire org. If it's compromised, everything is compromised. Treat it like a break-glass account.
  • The email address for each account must be unique. Use plus-addressing: aws+logarchive@company.com, aws+security@company.com, aws+network@company.com. This lets you use one inbox while satisfying the uniqueness constraint.
  • Pick your Control Tower home region carefully - it cannot be changed later. If you pick wrong, you'll need to decommission and re-create the entire landing zone.
  • Enable all features in Organizations, not just consolidated billing. Consolidated-billing-only mode doesn't support SCPs, tag policies, or other governance features. You'll hit a wall immediately.
  • Start with fewer accounts and grow. You can always split workloads into new accounts later. Merging accounts back together is not possible.

Key Insights

  • Multi-account is not optional for enterprise. A single AWS account with everything in it is a security and blast-radius nightmare. The multi-account pattern is AWS's recommended approach and every mature AWS deployment uses it.
  • Control Tower saves 2-4 weeks of setup. It encodes AWS best practices and keeps your landing zone in a known-good state. You can always customize after the baseline is in place.
  • SCPs are your guardrails - they prevent things, they don't enable things. An SCP cannot grant permissions. It can only restrict what IAM policies are allowed to do. Think of them as a ceiling, not a floor.
  • The Log Archive and Security accounts are non-negotiable. Centralized logging and a dedicated security account are foundational to any enterprise deployment. Don't skip them to "move fast" - you'll regret it within months.

  • Your existing GCP folder/project structure gives you the mental model. The concepts translate directly - AWS just uses different names and slightly different mechanics.

Service Control Policies (SCPs)

SCPs are organization-level policies that set the maximum permissions for accounts in your organization. They don't grant any permissions - they only restrict what IAM policies can do. If an SCP denies an action, no IAM policy in any affected account can allow it - not even the account root user.

They attach to OUs or individual accounts and cascade downward. An account's effective permissions are the intersection of its SCPs and its IAM policies.

AWS Control Tower

Control Tower is a managed service that sets up and governs a secure, multi-account AWS environment. It builds on top of AWS Organizations, AWS SSO (IAM Identity Center), and AWS Config to create a "landing zone" - a well-architected baseline environment.

It comes with preventive guardrails (SCPs) and detective guardrails (Config rules) out of the box. Think of it as the fast-track to a production-ready org structure.

Account Factory for Terraform (AFT)

AFT is a Terraform module that automates account provisioning in Control Tower. Instead of clicking through the console, you define accounts in code and AFT handles creation, baseline configuration, and OU placement.

It uses a GitOps pipeline - push account definitions to a repo, and AFT provisions them with your specified customizations, networking, and guardrails. It's the infrastructure-as-code approach to account vending.

Organizational Units (OUs)

OUs are containers within AWS Organizations that group accounts together. They're the primary mechanism for applying governance at scale - attach an SCP to an OU and it affects every account inside it, including nested sub-OUs.

Best practice is to organize by function (Security, Infrastructure, Workloads) rather than by team. Keep nesting shallow - two to three levels maximum - to keep policy inheritance predictable.

AWS Organizations

AWS Organizations is the service that lets you centrally manage multiple AWS accounts. The management account (formerly called "master account") is the root of the hierarchy and has special privileges - it can create accounts, attach SCPs, and manage billing for the entire org.

Never run workloads in the management account. Its elevated privileges make it a high-value target. Keep it locked down with MFA and minimal access.

Identity & Access Management

Your Entra ID is already the source of truth - now extend it to AWS without creating a third identity silo

IAM Identity Center Recommended

Formerly AWS SSO. This is your single pane of glass for human access to all AWS accounts. Connects to Entra ID via SAML/SCIM. Like what Entra already does for Azure - but for AWS. One portal, one login, every account.

Permission Sets

Templates that define what a user can do when they access an account. They become IAM Roles in each account. Similar concept to Azure RBAC role assignments but deployed per-account. You create them once, assign them to many accounts.

IAM Roles for Services

Workloads (EC2, Lambda, EKS pods) assume IAM Roles. No service account keys like GCP. No managed identity exactly like Azure - but the same zero-credential concept. Roles are the universal mechanism for both humans and machines.

Identity Federation Flow

Federation Setup

Recommended approach. This gives you centralized management, automatic user/group sync via SCIM , a single access portal, and native CLI integration. If you're starting fresh, this is the only path worth taking.

IAM Identity Center is automatically enabled if you used Control Tower (Tab 1). If not, enable it manually in your chosen home region.

  • Go to IAM Identity Center in the AWS Console (management account)
  • Choose the same region as your Control Tower home region
  • The service is free - there's no cost for Identity Center itself
One region only. IAM Identity Center can only be enabled in a single region. This cannot be changed later without disabling and re-enabling (which deletes all assignments). Choose wisely.

By default, Identity Center uses its own built-in directory. You need to switch it to use Entra ID as the external identity provider.

  1. In IAM Identity Center, go to Settings > Identity source > Actions > Change identity source
  2. Select External identity provider
  3. Download the IAM Identity Center SAML metadata file
  4. In Entra ID: Go to Enterprise Applications > New Application, search for "AWS IAM Identity Center"
  5. Upload the SAML metadata from AWS, configure the SAML SSO settings
  6. Back in AWS: Upload the Entra ID Federation metadata XML to complete the link

Test the SAML connection before proceeding. Both consoles have a "Test" button.

▼

Permission Sets define what users can do when they access an account. They're templates - when assigned, they create IAM Roles in each target account. Here are two real-world examples:

Admin access for your platform team:

{ "Name": "AdministratorAccess", "Description": "Full admin for platform team", "ManagedPolicies": ["arn:aws:iam::aws:policy/AdministratorAccess"], "SessionDuration": "PT4H", "Tags": [{"Key": "Team", "Value": "Platform"}] }

Developer access - broad compute, no IAM or billing:

{ "Name": "DeveloperAccess", "Description": "Dev access - no IAM or billing", "InlinePolicy": { "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", "Action": ["ec2:*", "s3:*", "lambda:*", "logs:*", "cloudwatch:*", "ecs:*", "ecr:*"], "Resource": "*" }, { "Effect": "Deny", "Action": ["iam:*", "organizations:*", "account:*"], "Resource": "*" } ] }, "SessionDuration": "PT8H" }

Note: 4-hour sessions for admin roles, 8-hour for dev roles. This is intentional - shorter sessions for higher-privilege access.

This is where it all comes together. You assign an Entra group to a specific AWS account with a specific Permission Set. Members of that group can then access that account with those permissions.

# Assign an Entra group to an account with a Permission Set aws sso-admin create-account-assignment \ --instance-arn arn:aws:sso:::instance/ssoins-xxx \ --target-id 123456789012 \ --target-type AWS_ACCOUNT \ --permission-set-arn arn:aws:sso:::permissionSet/ssoins-xxx/ps-xxx \ --principal-type GROUP \ --principal-id "entra-group-id-here"

Best practice: Always assign to groups, never to individual users. Manage membership in Entra, let SCIM sync handle the rest.

Developers add SSO profiles to their ~/.aws/config file. Each profile points to a specific account + role combination via the shared SSO session.

# ~/.aws/config [profile prod-admin] sso_session = my-sso sso_account_id = 123456789012 sso_role_name = AdministratorAccess region = eu-west-1 [profile dev-readonly] sso_session = my-sso sso_account_id = 234567890123 sso_role_name = ReadOnlyAccess region = eu-west-1 [sso-session my-sso] sso_start_url = https://my-company.awsapps.com/start sso_region = eu-west-1 sso_registration_scopes = sso:account:access

Then developers just run:

# Opens browser for Entra SSO - same flow as `az login` or `gcloud auth login` aws sso login --profile prod-admin # All subsequent commands use temporary credentials aws s3 ls --profile prod-admin aws ec2 describe-instances --profile dev-readonly
Legacy approach. You can federate directly to IAM roles in each account via SAML without Identity Center, but this doesn't scale. Each account needs its own identity provider configuration and trust policy. Use this only if you have a specific reason Identity Center doesn't work (e.g., GovCloud, China regions, or single-account setups).

Root User - Management Account

Store credentials in a physical safe or hardware-backed secret vault. Enable MFA with a hardware security key (YubiKey). This is your absolute last resort. Never use root for daily operations.

IAM User - Emergency Admin

Create a dedicated IAM user with MFA in the management account. This doesn't depend on Entra or SSO being available. Store credentials alongside root credentials. Rotate regularly even if never used.

Emergency admin IAM policy:

// Attach to the break-glass IAM user // Requires MFA for all actions { "Version": "2012-10-17", "Statement": [ { "Effect": "Allow", "Action": "*", "Resource": "*", "Condition": { "Bool": { "aws:MultiFactorAuthPresent": "true" } } } ] }

CloudTrail alert - detect any root login immediately:

// EventBridge rule pattern - triggers SNS/PagerDuty/Slack on root login { "source": ["aws.signin"], "detail-type": ["AWS Console Sign In via CloudTrail"], "detail": { "userIdentity": { "type": ["Root"] } } }
Test your break-glass procedure. At least quarterly, verify that emergency credentials work and that the MFA devices are functional. A break-glass path you've never tested is a break-glass path that won't work when you need it.
Coming from GCP: You're used to gcloud auth login authenticating you everywhere. In AWS, aws sso login is the equivalent - it gets you temporary credentials scoped to a specific account + role. The big difference: no persistent service account keys. AWS roles provide temporary credentials always.

Coming from Azure: Entra ID remains your IdP. You're just adding AWS as another Enterprise Application. Your conditional access policies, MFA requirements, and group memberships all flow through. It's the same pattern as adding any SaaS app to Entra.
Gotchas to watch for:
  • Single region lock-in: IAM Identity Center can only be enabled in ONE region. Choose wisely - it should match your Control Tower home region.
  • SCIM sync delay: Provisioning from Entra has a ~40 minute sync cycle. New group memberships aren't instant. Plan for this during onboarding.
  • Permission Set limits: Default limit is 20 Permission Sets per Identity Center instance. You can request an increase, but design your sets to be reusable rather than per-team.
  • Session duration matters: Short sessions for prod (4hr), longer for dev (8-12hr). Don't set everything to 12hr - the blast radius of a compromised session scales with duration.
  • No long-lived keys: IAM users with long-lived access keys should be avoided entirely. Use IAM roles + Identity Center for humans, IAM roles for services. If you find yourself creating access keys, you're doing it wrong.

SCIM (System for Cross-domain Identity Management)

SCIM is an open standard protocol for automating the exchange of user identity information between identity domains (like Entra ID) and service providers (like AWS Identity Center).

  • What it does: Automatically creates, updates, and deactivates user accounts and group memberships in AWS when changes happen in Entra ID.
  • Why it matters: Without SCIM, you'd have to manually create users in Identity Center or rely on just-in-time provisioning (which only creates users on first login and doesn't sync group changes).
  • How it works: Entra ID periodically pushes changes to the Identity Center SCIM endpoint. It syncs user attributes (name, email) and group memberships.
  • Key limitation: The sync cycle is approximately 40 minutes. It's not real-time. For urgent changes, you can trigger a manual sync from the Entra portal.

Azure equivalent: It's the same SCIM protocol Azure uses for provisioning to other SaaS apps. If you've set up SCIM provisioning for any Enterprise Application in Entra, this is identical.

Permission Set

A Permission Set is a template stored in IAM Identity Center that defines a collection of IAM policies. When you assign a Permission Set to a user/group on an AWS account, Identity Center creates an IAM Role in that account with those policies attached.

  • Composition: Can include AWS managed policies, customer managed policies, inline policies, and a permissions boundary.
  • Session duration: Each Permission Set has a configurable session duration (1-12 hours). This controls how long the temporary credentials last.
  • Scaling: You create a Permission Set once and assign it to as many account + group combinations as needed. It deploys consistently.
  • Default limit: 20 per Identity Center instance (can be increased via support request).

Azure equivalent: Think of it as an Azure RBAC role definition. Assigning a Permission Set to a group on an account is like assigning a role to a group on a subscription.

GCP equivalent: Similar to a custom IAM role, but with the added behavior of being automatically deployed to each account (project) you assign it to.

STS Temporary Credentials

AWS Security Token Service (STS) issues short-lived credentials consisting of an access key ID, a secret access key, and a session token. These expire automatically after a configured duration.

  • Components: Access Key ID (starts with ASIA), Secret Access Key, and Session Token. All three are required for API calls.
  • Duration: Configurable from 1 to 12 hours depending on the role and the Permission Set session duration setting.
  • No rotation needed: Because they expire automatically, there's no key rotation problem. Compare this to long-lived IAM access keys or GCP service account keys which must be manually rotated.
  • Transparent: When you use aws sso login, the CLI handles all the STS token exchange behind the scenes. You don't interact with STS directly.

Azure equivalent: Similar to the OAuth tokens you get from az login, but issued per-account and per-role rather than per-tenant.

GCP equivalent: Like the access tokens from gcloud auth print-access-token, which also expire after 1 hour by default.

AssumeRole

AssumeRole is the AWS API call that lets an entity (user, service, or another role) obtain temporary credentials for a different IAM Role. It's the universal mechanism for cross-account access and privilege escalation in AWS.

  • How it works: The caller provides the target role's ARN. STS validates the trust policy on that role, checks permissions, and returns temporary credentials.
  • Trust policies: Every IAM Role has a trust policy that specifies who is allowed to assume it. This is the access control gate.
  • For humans: Identity Center calls AssumeRole on your behalf when you select an account + Permission Set. You don't call it directly.
  • For services: EC2 instances, Lambda functions, and EKS pods assume roles automatically via instance profiles, execution roles, and IRSA/Pod Identity respectively.
  • Chaining: A role can assume another role (role chaining), enabling patterns like hub-and-spoke access across accounts.

Azure equivalent: No direct equivalent. The closest concept is using a managed identity to access resources in another subscription, but AssumeRole is more explicit and flexible.

GCP equivalent: Similar to gcloud iam service-accounts --impersonate (service account impersonation), but used universally for both humans and services.

Delete Default VPCs

Security requirement: Default VPCs come with public subnets, an Internet Gateway, and routes that allow public IP assignment. Delete them in every region across all accounts. This should be part of your account vending process.
delete-default-vpcs.sh 
#!/bin/bash
# Delete default VPCs in all regions
for region in $(aws ec2 describe-regions --query 'Regions[].RegionName' --output text); do
  vpc_id=$(aws ec2 describe-vpcs --region $region \
    --filters "Name=is-default,Values=true" \
    --query 'Vpcs[0].VpcId' --output text)
  if [ "$vpc_id" != "None" ]; then
    echo "Deleting default VPC $vpc_id in $region"
    # Delete subnets, IGW, then VPC
    aws ec2 delete-vpc --vpc-id $vpc_id --region $region
  fi
done

Transit Gateway - Your Network Hub

Transit Gateway is the central routing point for all VPC-to-VPC, VPC-to-on-prem, and VPC-to-other-cloud traffic within a region. If you're familiar with Azure, think of it as the Virtual WAN Hub - but simpler and with explicit route table management.

Key difference from Azure: In Azure, you attach VNets to a vWAN Hub and routing is largely automatic. In AWS, you attach VPCs to a Transit Gateway and manage route tables explicitly. This gives you more control over traffic segmentation.

Route Table Segmentation

Separate route tables on the Transit Gateway give you network-level isolation between environments. Production workloads cannot reach staging, and vice versa - enforced at the network layer, not just security groups.

Direct Connect - Leveraging Your Equinix Presence

Since you already have cross-connects in Equinix for GCP Cloud Interconnect and Azure ExpressRoute, adding AWS Direct Connect is straightforward - it is another cross-connect in the same facility. Same Equinix port, same rack, same cabling vendor.

1

Order a Direct Connect Connection

+

In the AWS Console (or via Terraform), create a Direct Connect connection specifying your Equinix facility. AWS will provide a Letter of Authorization (LOA) that you give to Equinix to provision the cross-connect. Since you already have infrastructure in the facility, this is just adding another cable.

Choose your speed: 1Gbps or 10Gbps dedicated connection. For enterprise workloads, 10Gbps is typical. You can also use a Hosted Connection via an Equinix partner for sub-1Gbps speeds.

2

Create a Direct Connect Gateway

+

The Direct Connect Gateway is a global resource - it is not tied to a region. It acts as the bridge between your physical connection and your Transit Gateways in multiple regions. One DX Gateway can connect to Transit Gateways in eu-west-1, us-east-1, and any other region simultaneously.

3

Associate with Transit Gateway

+

Associate the Direct Connect Gateway with your Transit Gateway in each region. You specify "allowed prefixes" - these are the CIDR ranges that will be advertised from AWS to your on-premises network via BGP.

4

Create a Transit VIF (Virtual Interface)

+

Create a Transit Virtual Interface on the physical connection, pointed at the Direct Connect Gateway. This is where BGP peering happens. Use your on-premises ASN (the same one your Palo Alto advertises to GCP and Azure) for consistency.

Why Transit VIF (not Private VIF)? Transit VIFs connect to Transit Gateways, which gives you transitive routing to all attached VPCs. Private VIFs connect directly to a single VPC - not what you want in a hub-spoke model.

Palo Alto VM-Series Integration via GWLB

Since you already run Palo Alto firewalls on-premises and likely in your other clouds, the AWS pattern is straightforward: deploy PA VM-Series instances behind a Gateway Load Balancer (GWLB) in a dedicated Inspection VPC. All traffic routed through the Transit Gateway default route passes through your Palo Alto instances - same vendor, same policies, same management plane.

How It Works

  1. Transit Gateway sends all default-route traffic (0.0.0.0/0) to the Inspection VPC attachment
  2. The Inspection VPC has a GWLB Endpoint in the TGW-facing subnet
  3. GWLB distributes traffic (using GENEVE encapsulation) across PA VM-Series instances in multiple AZs
  4. PA VM-Series inspects, filters, and returns the traffic via the same GWLB
  5. Inspected traffic routes back through the Transit Gateway to the destination VPC

Transit Gateway

  |-- GWLB Endpoint (in TGW subnet)
  |-- Gateway Load Balancer
  |-- PA VM-Series Instance 1 (AZ-a)
  +-- PA VM-Series Instance 2 (AZ-b)
  |
  v (inspected traffic returns via GWLB)
Transit Gateway -> destination VPC

Critical Considerations

  • CIDR ranges cannot overlap with your GCP and Azure ranges if you are routing between them via Direct Connect. Plan this FIRST before deploying any VPCs.
  • Transit Gateway has per-attachment and per-GB charges. Each VPC attachment costs roughly $0.05/hr (~$36/month) plus $0.02/GB of data processed. Budget for this - it adds up across many accounts.
  • Direct Connect takes 2-4 weeks to provision. Even in Equinix with existing infrastructure, the LOA process and cross-connect installation take time. Start this early - it is on the critical path.
  • GWLB adds latency of approximately 1-2ms per hop. Acceptable for security inspection, but be aware for latency-sensitive workloads. You can use TGW route tables to bypass inspection for specific trusted flows if needed.
  • Do not create public subnets unless absolutely necessary. The only valid reasons are Application Load Balancers and NAT Gateways. Default to private subnets for everything else.

Transit Gateway

What it is: A regional network hub managed by AWS that enables transitive routing between VPCs, VPN connections, and Direct Connect gateways. Think of it as a cloud-native router.

Azure equivalent: Virtual WAN Hub - but with more explicit route table control. In Azure, vWAN handles much of the routing automatically. In AWS, you define route tables and associations explicitly, giving you finer-grained segmentation.

GCP equivalent: No direct equivalent. GCP uses a global VPC model with VPC peering, which is fundamentally different. The closest concept would be a combination of VPC peering and Cloud Router.

Key behaviors:

  • Regional resource - one per region, shared across accounts via RAM
  • Supports up to 5,000 attachments per TGW
  • Route tables control which attachments can communicate
  • Supports equal-cost multi-path (ECMP) routing for Direct Connect
  • Charged per attachment per hour plus per GB of data processed

Direct Connect Gateway

What it is: A globally available resource that acts as an intermediary between your Direct Connect connection (physical) and your Transit Gateways or VPCs (logical). It decouples the physical connection from the regional resources.

Why it matters: Without a DX Gateway, a Direct Connect connection can only reach resources in the region where the connection terminates. With a DX Gateway, a single physical connection in Equinix London can reach Transit Gateways in eu-west-1, us-east-1, ap-southeast-1, and any other region.

Azure equivalent: ExpressRoute circuit with Global Reach. The DX Gateway concept is similar to how an ExpressRoute circuit can connect to multiple VNet Gateways across regions.

Key detail: You associate "allowed prefixes" when linking a DX Gateway to a Transit Gateway. These are the CIDRs advertised from AWS to your on-prem routers via BGP. Plan these carefully to avoid advertising overlapping ranges.

Gateway Load Balancer (GWLB)

What it is: A load balancer specifically designed for deploying third-party network appliances (firewalls, IDS/IPS, DLP) transparently in the traffic path. It uses GENEVE encapsulation (port 6081) to send traffic to appliances while preserving the original source and destination.

Why not a regular NLB? Network Load Balancers terminate connections. GWLB passes traffic transparently - the firewall sees the original packets, not load-balancer-originated traffic. This is essential for stateful firewall inspection.

How it works with Palo Alto:

  • Traffic arrives at a GWLB Endpoint (deployed in the subnet where traffic enters)
  • GWLB encapsulates the traffic in GENEVE and forwards to a PA VM-Series instance
  • PA inspects, applies policies, and returns the traffic via GENEVE
  • GWLB de-encapsulates and forwards to the destination

Azure equivalent: Azure Gateway Load Balancer - same concept, same vendor support, same GENEVE encapsulation.

RAM (Resource Access Manager)

What it is: An AWS service that lets you share resources across accounts within your Organization. Instead of creating a Transit Gateway in every account, you create one in the Network account and share it via RAM to all other accounts.

What you can share: Transit Gateways, subnets, Route 53 Resolver rules, License Manager configurations, and more. For networking, the key shares are Transit Gateways and (optionally) subnets.

How it works:

  • Create a Resource Share in RAM
  • Add the resource (e.g., Transit Gateway ARN)
  • Add principals (Organization ID, OU IDs, or specific account IDs)
  • If sharing within an Organization with sharing enabled, acceptance is automatic

Azure equivalent: No direct equivalent needed - Azure resources like Virtual WAN Hubs can be referenced cross-subscription natively. RAM fills a gap created by AWS's strict account-level resource isolation.

VIF (Virtual Interface)

What it is: A Virtual Interface is a logical connection created on top of a physical Direct Connect connection. It is where BGP peering configuration lives - VLAN tag, BGP ASN, IP addresses, and authentication.

Three types:

  • Transit VIF: Connects to a Direct Connect Gateway associated with Transit Gateways. This is what you want for hub-spoke architectures - one VIF reaches all VPCs via the TGW.
  • Private VIF: Connects to a Direct Connect Gateway or directly to a single VPC's Virtual Private Gateway. Use this only for isolated VPC access.
  • Public VIF: Provides access to AWS public services (S3, DynamoDB, etc.) over Direct Connect rather than the internet. Useful but not required if you use VPC Endpoints.

Azure equivalent: ExpressRoute peering configuration. A Transit VIF is similar to Azure Private Peering on an ExpressRoute circuit connected to a Virtual WAN Hub.

Key setting: Set MTU to 8500 (jumbo frames) if your on-prem network supports it. This reduces fragmentation and improves throughput.

GuardDuty - Org-Wide Threat Detection

▼

Enable org-wide from your Security Tooling account as delegated admin. GuardDuty automatically analyzes CloudTrail management events, VPC Flow Logs, and DNS query logs across every account in your organization.

# In management account - delegate GuardDuty admin
aws guardduty create-detector --enable
aws guardduty enable-organization-admin-account \
  --admin-account-id SECURITY_ACCOUNT_ID

# In security account - enable for all org accounts
aws guardduty create-members \
  --detector-id DETECTOR_ID \
  --account-details '[{"AccountId":"111","Email":"a@co.com"}]'
aws guardduty update-organization-configuration \
  --detector-id DETECTOR_ID \
  --auto-enable
Cost estimate: ~$1/GB for CloudTrail analysis, ~$1/GB for VPC Flow Logs analysis. Budget roughly $500-2000/mo for a mid-size deployment (10-20 accounts, moderate traffic). GuardDuty has a 30-day free trial - use it to establish your baseline cost before committing.

CloudTrail - Organization Trail

▼

A single organization trail covers all accounts automatically - new accounts are included the moment they join the org. No per-account configuration needed.

# Terraform - Organization CloudTrail
resource "aws_cloudtrail" "org_trail" {
  name                       = "org-trail"
  s3_bucket_name             = aws_s3_bucket.cloudtrail_logs.id
  is_organization_trail      = true
  is_multi_region_trail      = true
  enable_log_file_validation = true

  cloud_watch_logs_group_arn = "${aws_cloudwatch_log_group.cloudtrail.arn}:*"
  cloud_watch_logs_role_arn  = aws_iam_role.cloudtrail_cw.arn

  event_selector {
    read_write_type           = "All"
    include_management_events = true

    data_resource {
      type   = "AWS::S3::Object"
      values = ["arn:aws:s3"]
    }
  }

  tags = { Name = "organization-trail" }
}

Gotcha Data events (S3 object-level, Lambda invocations) cost extra and generate massive volume. Start with management events only, then selectively enable data events for high-value buckets.

This is the critical section. You already have a Palo Alto SIEM / Cortex XSIAM that ingests GCP and Azure security data. The goal here is to add AWS as another data source - not replace your existing tooling.
A

S3 + Cortex XSIAM/XSOAR Collector (Recommended for Bulk)

▼

Best if you're using Cortex XSIAM. The XSIAM S3 collector polls your centralized S3 buckets on a schedule and ingests everything.

  • CloudTrail org trail -> S3 (Log Archive account) -> Cortex XSIAM S3 collector
  • VPC Flow Logs -> S3 -> Cortex XSIAM S3 collector
  • GuardDuty findings -> EventBridge -> Lambda -> Cortex XSIAM API

S3 bucket policy for cross-account SIEM access:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowSIEMRead",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::SIEM_ACCOUNT:role/cortex-xsiam-collector"
      },
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::centralized-cloudtrail-logs",
        "arn:aws:s3:::centralized-cloudtrail-logs/*"
      ]
    }
  ]
}

Gotcha Make sure the SIEM IAM role uses an external ID in its trust policy. Without it, any AWS account that knows your role ARN could assume it (confused deputy problem).

B

Kinesis Data Firehose -> SIEM (Near Real-Time)

▼

For near real-time log delivery with lower latency than S3 polling:

  • CloudWatch Logs (subscription filter) -> Kinesis Firehose -> HTTP endpoint (SIEM)
  • Lower latency than S3 polling but more complex to set up and maintain
  • Firehose handles batching, compression, and retry logic

Use this when your SIEM SLA requires sub-minute log delivery. For most organizations, S3-based ingestion with 5-15 minute polling intervals is sufficient.

C

EventBridge + Lambda for Critical Alerts (Instant)

▼

For high-priority findings only - this gets critical GuardDuty alerts to your SIEM in seconds, not minutes.

Lambda function - forward GuardDuty findings to SIEM:

# Lambda function: forward GuardDuty findings to SIEM
import json
import urllib3

def handler(event, context):
    finding = event['detail']
    severity = finding['severity']

    # Only forward HIGH and CRITICAL
    if severity >= 7.0:
        siem_payload = {
            "source": "aws-guardduty",
            "severity": severity,
            "title": finding['title'],
            "description": finding['description'],
            "account_id": finding['accountId'],
            "region": finding['region'],
            "resource": finding['resource'],
            "type": finding['type']
        }

        http = urllib3.PoolManager()
        http.request('POST',
            'https://siem.company.com/api/v1/ingest',
            body=json.dumps(siem_payload),
            headers={'Content-Type': 'application/json',
                     'Authorization': 'Bearer ${SIEM_TOKEN}'})

EventBridge rule - filter for high-severity GuardDuty findings:

{
  "source": ["aws.guardduty"],
  "detail-type": ["GuardDuty Finding"],
  "detail": {
    "severity": [{ "numeric": [">=", 7] }]
  }
}
Recommended approach: Use S3 for bulk log ingestion (CloudTrail, VPC Flow Logs, Config) and EventBridge + Lambda for real-time critical alerts (GuardDuty HIGH/CRITICAL). This matches how your GCP SCC and Azure Defender findings likely flow to your SIEM today - bulk logs via a collector, critical alerts via webhook/API.
Day-1 compliance posture. These are the guardrails you deploy immediately - before any workloads land. They prevent the most common security mistakes and ensure your AWS environment starts secure.

SCP - Prevent Disabling Security Services

▼

This SCP prevents anyone - including account administrators - from disabling GuardDuty, CloudTrail, Config, or Security Hub. Attach it at the root OU so it applies to every account.

Region restriction SCP is covered in the Org & Accounts tab - this complements it.

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyDisablingSecurityServices",
      "Effect": "Deny",
      "Action": [
        "guardduty:DeleteDetector",
        "guardduty:DisassociateFromMasterAccount",
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail",
        "config:StopConfigurationRecorder",
        "config:DeleteConfigurationRecorder",
        "securityhub:DisableSecurityHub"
      ],
      "Resource": "*",
      "Condition": {
        "ArnNotLike": {
          "aws:PrincipalARN": [
            "arn:aws:iam::*:role/AWSControlTowerExecution",
            "arn:aws:iam::*:role/SecurityAuditRole"
          ]
        }
      }
    }
  ]
}

Gotcha The ArnNotLike condition exempts Control Tower's execution role and your security audit role. Without this exemption, you can't make legitimate changes to security services even from the management account.

2

S3 Block Public Access (Org-Wide)

▼

Block public access to all S3 buckets at the organization level. This is a single command that prevents the most common AWS security mistake - accidentally public S3 buckets.

# Block public access at the account level (run for each account)
aws s3control put-public-access-block \
  --account-id MGMT_ACCOUNT_ID \
  --public-access-block-configuration \
    BlockPublicAcls=true,\
    IgnorePublicAcls=true,\
    BlockPublicPolicy=true,\
    RestrictPublicBuckets=true

Apply this to every account via Control Tower guardrails or via your account vending pipeline. Individual teams should never need public S3 buckets - use CloudFront with Origin Access Control for public content delivery instead.

4

IMDSv2 Enforcement (Prevent SSRF Credential Theft)

▼

The Instance Metadata Service (IMDS) is how EC2 instances get their IAM credentials. IMDSv1 is vulnerable to SSRF attacks - an attacker who can make HTTP requests from your instance can steal its credentials. IMDSv2 requires a token-based approach that blocks this attack vector.

SCP to enforce IMDSv2 on all new instances:

{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "RequireIMDSv2",
    "Effect": "Deny",
    "Action": "ec2:RunInstances",
    "Resource": "arn:aws:ec2:*:*:instance/*",
    "Condition": {
      "StringNotEquals": {
        "ec2:MetadataHttpTokens": "required"
      }
    }
  }]
}

Gotcha Some older AMIs and SDKs don't support IMDSv2 out of the box. Test your workloads first. Most modern AWS SDKs (2019+) handle IMDSv2 automatically.

Amazon GuardDuty

GuardDuty is a managed threat detection service that continuously monitors your AWS accounts and workloads for malicious activity. It analyzes CloudTrail management and data events, VPC Flow Logs, DNS query logs, and optionally EKS audit logs and S3 data events.

Findings are categorized by severity (LOW 1-3.9, MEDIUM 4-6.9, HIGH 7-8.9, CRITICAL 9-10) and include actionable details about the threat, affected resource, and recommended remediation.

  • No agents to install - it reads existing log sources
  • Machine learning detects anomalous behavior (unusual API calls, credential exfiltration, crypto mining)
  • Delegated admin model - manage all accounts from Security Tooling account
  • 30-day free trial to establish cost baseline

AWS Security Hub

Security Hub provides a comprehensive view of your security posture across all AWS accounts. It aggregates, organizes, and prioritizes findings from GuardDuty, Inspector, IAM Access Analyzer, Firewall Manager, and Config.

Key features:

  • Compliance standards - CIS AWS Foundations Benchmark, AWS Foundational Security Best Practices, PCI DSS
  • Security score - percentage of passed controls, similar to Azure Secure Score
  • Cross-account aggregation - all findings from all accounts in one dashboard
  • Automated response - EventBridge integration for auto-remediation workflows
  • ASFF format - standardized finding format that makes SIEM integration straightforward

AWS Config

AWS Config continuously records the configuration of your AWS resources and evaluates them against desired configurations. Think of it as GCP Asset Inventory combined with a compliance rules engine.

  • Configuration recording - tracks every change to every resource (who changed what, when)
  • Config Rules - evaluates resources against compliance policies (managed rules or custom Lambda-based rules)
  • Conformance Packs - pre-built collections of Config Rules for common compliance frameworks (CIS, PCI, HIPAA)
  • Aggregator - collects Config data from all accounts into a central view in your Security account
  • Timeline - visualize the full configuration history of any resource

CloudTrail Organization Trail

An organization trail is a CloudTrail configuration that automatically logs API activity from all accounts in your AWS Organization. When a new account joins the org, it's automatically included - no manual setup needed.

  • Management events - API calls that manage your AWS resources (CreateInstance, DeleteBucket, etc.) - included by default
  • Data events - API calls within resources (S3 GetObject, Lambda Invoke) - optional, higher volume and cost
  • Log file validation - cryptographic hash chain that proves logs haven't been tampered with
  • Multi-region - captures API calls in all regions, not just the home region
  • Logs land in the Log Archive account's S3 bucket - set lifecycle policies to manage retention and cost

Amazon EventBridge

EventBridge is a serverless event bus that routes events between AWS services, SaaS applications, and your own applications. For security, it's the primary mechanism for real-time event routing.

  • Event rules - filter events by source, type, and content (e.g., "GuardDuty findings with severity >= 7")
  • Targets - route matching events to Lambda, SNS, SQS, Step Functions, or HTTP endpoints
  • Cross-account - forward events from member accounts to the Security Tooling account's event bus
  • Near real-time - events are delivered in seconds, not minutes
  • Every AWS service emits events to EventBridge - it's the backbone of event-driven security automation

IMDSv2 (Instance Metadata Service v2)

The Instance Metadata Service (IMDS) is how EC2 instances discover their identity, IAM credentials, and configuration. IMDSv1 uses a simple HTTP GET to a well-known IP (169.254.169.254) - making it trivially exploitable via SSRF vulnerabilities in web applications.

IMDSv2 adds a session token requirement:

  • Step 1 - PUT request to get a session token (with a TTL you specify)
  • Step 2 - GET request with the token in a header
  • The PUT request uses a hop limit of 1, preventing remote SSRF exploitation
  • Enforcing IMDSv2 via SCP ensures no one can launch instances with the vulnerable v1 endpoint
  • Capital One's 2019 breach was enabled by IMDSv1 - this is a real-world risk

Account Factory for Terraform (AFT)

AFT is an AWS-maintained Terraform pipeline that automates account provisioning through Control Tower. Instead of manually clicking through the Account Factory console, you define accounts as Terraform configurations in a Git repository.

When you push a new account definition, AFT automatically:

  • Creates the account via Control Tower
  • Places it in the correct OU
  • Applies account-level customizations (VPC setup, IAM roles, security baseline)
  • Runs any global customizations (standard tags, Config rules)

AFT deploys into a dedicated account and uses CodePipeline + Step Functions under the hood. It requires Terraform experience but pays off quickly once you're creating more than a handful of accounts.

Site-to-Site VPN

AWS Site-to-Site VPN creates encrypted IPsec tunnels between your on-premises network and AWS VPCs (or Transit Gateway). Each VPN connection provides two tunnels for redundancy across different AWS endpoints.

Key characteristics compared to Direct Connect:

  • Setup time: Minutes (vs. weeks for DX)
  • Bandwidth: Up to 1.25 Gbps per tunnel (vs. 1-100 Gbps for DX)
  • Latency: Variable, internet-dependent (vs. consistent low latency on DX)
  • Cost: ~$36/mo per connection + data transfer (vs. port fees for DX)
  • Encryption: Built-in IPsec (DX requires MACsec or overlay VPN for encryption)

Best used as a temporary bridge while Direct Connect is being provisioned, and then retained as a backup path for DX failover.

VPC Endpoints

VPC Endpoints let your resources access AWS services (S3, DynamoDB, ECR, STS, SSM, etc.) without sending traffic over the public internet. Traffic stays on the AWS private network.

Two types:

  • Gateway endpoints (S3, DynamoDB) - free, route-table based
  • Interface endpoints (everything else) - ~$7.30/mo per AZ + data processing. These create ENIs in your subnets with private IPs.

In a multi-account setup, you can centralize interface endpoints in a Shared Services VPC and share them via Private Hosted Zones, reducing cost and management overhead. This is particularly valuable when you have many accounts that all need access to the same set of AWS services.

NAT Gateway

NAT Gateways allow resources in private subnets to initiate outbound connections to the internet (for software updates, API calls, etc.) while preventing unsolicited inbound connections.

Pricing has two components that catch people off guard:

  • Hourly charge: ~$0.045/hr per gateway (~$32/mo) - predictable
  • Data processing: $0.045/GB - this is the surprise. High-throughput workloads can generate hundreds or thousands of dollars in NAT data charges.

Cost optimization strategies:

  • Use VPC Endpoints for AWS service traffic (avoids NAT entirely)
  • Consider NAT instances (EC2-based) for non-prod where availability isn't critical
  • Centralize NAT in the Network Hub and route through TGW (fewer gateways, but adds TGW data charges)
  • Monitor with VPC Flow Logs to identify top talkers

Cost Allocation Tags

Cost allocation tags are metadata key-value pairs attached to AWS resources that allow you to break down costs by team, project, environment, or any other dimension. They're your primary mechanism for chargeback and showback reporting.

Two types:

  • AWS-generated tags - automatically applied (e.g., aws:createdBy)
  • User-defined tags - you create and apply these (e.g., team, environment, project)

Critical setup steps:

  • Define a tagging standard before creating resources
  • Activate tags in the Billing console (they don't appear in Cost Explorer until activated)
  • Enforce tagging via SCPs or AWS Config rules (required-tags)
  • Tags are not retroactive in billing - untagged resources from before activation will remain untagged in cost reports

Recommended minimum tag set: Environment, Team, Project, CostCenter, ManagedBy (Terraform/manual/CDK).
